# 收单 AI 整体规划

## 一句话目标

建设一个面向企业内部收单场景的 Agent 平台，而不是若干离散聊天机器人。

## 当前统一口径

平台最新技术与架构口径如下：

- 运行时：`AgentScope`
- 服务框架：`FastAPI`
- 主数据库：`MySQL 8.0`
- 缓存与短状态：`Redis`
- 接入方式：`MCP + Internal Tools`
- 高风险流程：`Workflow + Approval`
- 注册发现：`Nacos`
- 部署方式：`Docker + Kubernetes`

平台核心目标已经从“Agent 接入平台”演进为：

- Agent 运行
- 任务实时展示
- 审计与审批治理
- 评估与持续优化

## 总体目标

平台围绕四类业务域展开：

- 商户 AI
- 代理商 AI
- 运营支持
- 数据支持

整体目标不是追求泛化聊天能力，而是追求：

- 业务问题可稳定解决
- 执行过程可展示
- 高风险流程可治理
- 结果可审计
- Agent 能力可持续优化

## 总体设计思路

平台建议分为五层：

- 渠道接入层
- 业务域 Agent 层
- AgentScope 运行层
- 能力层
- 平台治理底座层

其中能力层统一抽象为：

- `Skill`
- `Tool`
- `Workflow`
- `Knowledge`

治理底座统一承载：

- 审计
- 审批
- 评估
- 风险控制
- 注册发现
- 配置管理

## 平台主线

当前平台以 `任务` 为核心主线，而不是只以“聊天消息”为中心。

统一主线对象包括：

- `Contact`
- `Message`
- `Task`
- `Task Event`
- `Task Artifact`
- `Approval`
- `Audit`
- `Evaluation`

这意味着平台可以展示：

- 用户问题
- Agent 执行过程
- 工具/MCP 调用过程
- 审批过程
- 最终结果
- 评估结果

## 为什么以 AgentScope 为主

当前平台更适合 `AgentScope` 的原因是：

- 需要明确运行时边界
- 需要逐步接入多 Agent、Tool、Workflow
- 需要和任务事件流、评估链路打通
- 需要更强的企业内编排可控性

因此当前方向不再是“以 Agno 为主”，而是“以 AgentScope 为主运行时边界”。

## 数据底座规划

当前数据底座收敛为：

- 主事务库：`MySQL 8.0`
- 缓存与短状态：`Redis`

说明：

- 本地开发仍允许 `SQLite`
- 向量检索不再绑定 `pgvector`
- 后续建议将向量检索独立为外部能力

## 一期范围

一期建议聚焦三个闭环：

### 1. Agent 执行闭环

- 路由
- Agent 执行
- Tool / MCP 接入
- 审批接管

### 2. 任务展示闭环

- 任务主表
- 任务事件流
- 任务产物
- SSE 实时推送

### 3. 评估优化闭环

- 在线评估
- 维度评分
- 优化建议
- 后续调优入口

## 一期明确不做

- 不追求一步到位的完整多 Agent 群协作
- 不做完整 MCP 协议客户端全集
- 不做完整向量检索体系
- 不让 AI 绕过审批直接执行高风险动作

## 当前演进方向

下一阶段建议重点推进：

1. `AgentScope runtime` 真实化
2. MySQL 8.0 正式迁移体系
3. Redis 事件与状态职责
4. MCP 真正客户端执行
5. Workflow / Risk / Approval 深化

## 最终目标

最终交付的是一个：

- 可扩展
- 可治理
- 可展示
- 可审计
- 可持续优化

的企业级收单 Agent 平台。
