# 收单AI整体规划

## 一句话目标

建设一个面向收单业务的企业级智能服务平台，而不是零散地做几个聊天机器人。

## 总体目标

平台以 `Agno` 为主框架，围绕四条核心业务线展开：

- 商户AI
- 代理商AI
- 运营支持
- 数据支持

整体目标不是追求“AI很聪明”，而是追求：

- 业务问题能稳定解决
- 流程可控
- 结果可审计
- 能力可复用和持续扩展

## 业务域规划

### 商户AI

- 知识库问答
- 常见问题解答
- 问题分类与处理建议
- 经营分析初步解读

### 代理商AI

- 政策问答
- 常见问题处理
- 代理商经营分析
- 渠道支持知识服务

### 运营支持

- 风险处置分析
- 司法协查辅助
- 调额审核
- 解冻审核
- T0开通
- 入网审核
- 变更审核
- 商户合规支持
- 清算支持
- 差错处理

### 数据支持

- 直营销售数据助手
- 分公司数据助手
- 合规监管报表
- 合规对接与口径解释

## 总体设计思路

整体架构建议分成五层：

- 渠道层
- 业务域层
- Agent编排层
- 能力层
- 平台底座层

其中能力层需要明确拆分为：

- `Skill`：业务规则、SOP、处理方法、输出口径
- `Tool`：查询系统、调用接口、执行动作
- `Workflow`：审核、审批、回退、人工确认等确定性流程
- `Knowledge`：制度、FAQ、案例、监管口径、业务文档

## 为什么以 Agno 为主

这套体系对框架的要求不是单纯对话，而是要同时支持：

- Skills
- Workflow
- Approval/HITL
- RBAC
- MCP/Tool 接入
- Guardrails

从需求匹配度看，`Agno` 更适合“规则 + 工具 + 流程 + 审批”的企业内部场景。

## 平台底座

底座能力建议优先建设：

- 统一身份认证与 RBAC
- Skill Registry
- Tool Registry
- Workflow Registry
- 知识检索与权限过滤
- 审批中心
- 审计日志
- 模型网关
- 评测与观测

## 推荐技术栈

- Agent框架：`Agno`
- 服务框架：`FastAPI`
- 数据库：`PostgreSQL`
- 向量检索：`pgvector`
- 缓存与短状态：`Redis`
- 内部系统接入：`MCP + Internal Tools`
- 审批与人工介入：`Workflow + Approval/HITL`
- 部署方式：`Docker + Kubernetes`

## 扩展性原则

- 业务域独立扩展，互不强耦合
- 所有新能力统一抽象成 Skill、Tool、Workflow、Knowledge
- 高风险动作统一收口到审批和审计
- 规则尽量配置化，不写死在 Agent 内部
- 新场景像装模块，不像改系统

## 第一期范围

第一期建议优先做三条线：

- 商户AI：知识问答、问题分类、处理建议
- 运营支持：调额审核辅助、入网审核辅助、变更审核辅助
- 数据支持：直营销售数据助手、固定报表辅助

第一期同步建设的平台能力：

- 权限控制
- 审批流
- 审计日志
- Skill/Tool 基础注册能力
- 知识检索
- 基础观测与评测

## 第一期明确不做

- 不一次覆盖全部业务域
- 不做完全自治的多 Agent 协作
- 不做大规模自动写操作
- 不让 AI 替代人工审批
- 不做过于开放的自由分析型数据助手

## 演进路线

### 阶段一：MVP验证

- 跑通商户AI、运营支持、数据支持的代表场景
- 搭建权限、审批、审计、知识库底座

### 阶段二：平台扩展

- 接入代理商AI
- 扩展更多审核流程
- 完善 Skill、Tool、Workflow 治理和版本体系

### 阶段三：智能运营平台

- 新场景配置化接入
- 更多业务域协同
- 更强的经营分析和平台化运营能力

## 最终目标

最终交付的是一个：

- 可扩展
- 可治理
- 可审计
- 可复用
- 可持续演进

的企业级收单AI平台。
