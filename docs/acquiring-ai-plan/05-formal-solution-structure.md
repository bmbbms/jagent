# 正式方案文档结构

## 一、项目背景

收单业务涉及商户、代理商、运营、数据、合规等多角色协同，业务规则复杂、知识分散、流程依赖人工，存在知识获取成本高、处理口径不统一、审核效率不足、报表产出周期长等问题，需要建设统一的智能服务平台。

## 二、建设目标

- 覆盖商户AI、代理商AI、运营支持、数据支持四大核心业务域
- 支撑知识问答、问题处理、审核辅助、数据查询、报表生成等代表性场景
- 形成统一的技能、工具、工作流、知识管理机制
- 建立权限、审批、审计、评测等企业级平台底座
- 实现能力可复用、规则可治理、流程可追溯、平台可持续演进

## 三、总体建设思路

- 采用“分层架构 + 平台底座 + 业务模块化接入”的设计思路
- 所有能力统一抽象为 Skill、Tool、Workflow、Knowledge
- 高风险能力统一走 Workflow + Approval
- 新增场景按统一规范接入平台，而非重新开发独立系统

## 四、总体架构设计

整体架构包括：

- 渠道层
- 业务域层
- Agent编排层
- 能力层
- 平台底座层

设计原则：编排层负责调度，能力层负责执行，平台层负责治理。

## 五、技术路线与选型

- Agent框架：Agno
- 服务框架：FastAPI
- 数据库：PostgreSQL
- 向量检索：pgvector
- 缓存与短状态：Redis
- 内部系统接入：MCP + Internal Tools
- 审批与人工介入：Workflow + Approval/HITL
- 部署方式：Docker + Kubernetes

## 六、业务域规划

- 商户AI：知识问答、问题处理、经营分析
- 代理商AI：政策问答、问题处理、经营分析
- 运营支持：风险处置、司法协查、调额/解冻/T0/入网/变更审核、清算、差错处理
- 数据支持：销售数据助手、分公司数据助手、合规监管报表、合规对接

## 七、扩展性设计

重点建设机制：

- Skill Registry
- Tool Registry
- Workflow Registry
- Knowledge Scope
- Model Gateway
- Evaluation Layer

未来新增场景时，原则上只新增业务包、skills、tools、workflows 和评测用例，不改动平台主干。

## 八、第一期建设范围

第一期建议覆盖：

- 商户AI：知识问答、问题分类、处理建议
- 运营支持：调额审核辅助、入网审核辅助、变更审核辅助
- 数据支持：直营销售数据助手、固定报表辅助

同时建设：

- 统一登录与权限控制
- Skill/Tool 基础注册能力
- Workflow + Approval
- 审计日志
- 知识检索
- 基础监控与评测

## 九、第一期建设边界

- 不一次性覆盖全部业务域和全部业务场景
- 不建设完全自治的多 Agent 协同体系
- 不开放大规模自动写操作
- 不替代人工审批，仅作为辅助和提效
- 不建设复杂跨部门联动的全流程自动化

## 十、实施路径与演进路线

### 阶段一：MVP验证

- 跑通商户AI、运营支持、数据支持核心闭环
- 建设权限、审批、审计、知识库等底座能力

### 阶段二：平台扩展

- 接入代理商AI
- 覆盖更多运营审核场景
- 完善 Skill、Tool、Workflow 治理与版本管理

### 阶段三：智能运营平台

- 支持更多场景配置化接入
- 增强跨业务域协同
- 建设平台级运营看板与持续优化体系

## 十一、预期价值

- 提升知识获取和问题处理效率
- 提高审核标准化和一致性
- 缩短数据查询和报表生成时间
- 沉淀业务规则、制度口径和流程资产
- 建立统一智能化平台，降低后续扩展成本

## 十二、关键风险与保障建议

- 场景过散：第一期聚焦少量代表性闭环场景
- 规则与流程混杂：严格区分 Skill、Tool、Workflow、Knowledge
- 权限与审计不足：从第一期开始同步建设 RBAC、Approval、Audit
- 业务规则依赖个人经验：推动业务专家参与 Skill 和 Workflow 沉淀
- 效果难以评估：建立基础评测、反馈和持续优化机制

## 十三、结论

项目核心不是建设单点 AI 功能，而是打造一个面向收单业务的智能服务平台。平台以 Agno 为核心框架，以技能、工具、流程、知识分层为基础，以权限、审批、审计、评测为治理底座，逐步演进为可扩展、可治理、可审计、可持续优化的企业级收单AI平台。
