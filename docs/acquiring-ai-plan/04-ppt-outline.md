# 汇报提纲

## 1. 项目背景

- 收单业务场景复杂，涉及商户、代理商、运营、数据等多角色协同
- 当前知识分散、流程依赖人工、处理口径不统一
- 审核、问答、报表、问题处理存在效率和一致性瓶颈
- 需要建设统一的智能服务体系，提升效率与治理能力

## 2. 项目目标

- 建设面向收单业务的统一 AI 智能服务平台
- 优先覆盖商户AI、运营支持、数据支持等核心场景
- 实现知识问答、审核辅助、数据查询的能力闭环
- 形成可扩展、可治理、可审计的平台底座

## 3. 总体建设思路

- 不是建设多个孤立机器人，而是建设统一平台
- 采用“业务域 + 能力层 + 平台底座”分层设计
- 高风险场景优先走工作流和审批，不走完全自由对话
- 新增业务场景按模块接入，避免重复建设

## 4. 业务蓝图

- 商户AI：知识库、问题解答、问题处理、经营分析
- 代理商AI：知识库、问题解答、问题处理、经营分析
- 运营支持：风险处置、司法协查、业务审核、商户审核、清算支持、差错处理
- 数据支持：销售数据助手、分公司数据助手、合规监管报表、合规对接

## 5. 总体分层架构

- 渠道层：Web、企业微信、OpenAPI
- 业务域层：商户AI、代理商AI、运营支持、数据支持
- 编排层：Router Agent + Domain Agents
- 能力层：Skill / Tool / Workflow / Knowledge
- 平台底座：权限、审批、审计、评测、模型网关

## 6. 为什么这样分层

- Skill 负责业务规则和 SOP
- Tool 负责系统查询与执行动作
- Workflow 负责审核、审批、回退等确定性流程
- Knowledge 负责制度、FAQ、案例和口径依据
- 分层后更容易复用、治理、审计和扩展

## 7. 技术选型建议

- 主框架：Agno
- 服务框架：FastAPI
- 数据底座：PostgreSQL + pgvector + Redis
- 系统接入：MCP + Internal Tools
- 高风险流程：Workflow + Approval
- 部署方式：Docker + Kubernetes

## 8. 为什么以 Agno 为主

- 原生支持 Skills
- 支持 Workflow 和 Approval/HITL
- 支持 MCP、Guardrails、RBAC
- 更适合企业内部“规则 + 工具 + 流程”型场景
- 与收单业务中的审核、问答、报表场景匹配度高

## 9. 扩展性设计原则

- 业务域独立扩展，互不强耦合
- 新能力统一抽象为 Skill / Tool / Workflow / Knowledge
- 高风险动作统一收口到审批和审计
- 规则尽量配置化，不写死在 Agent 内部
- 新场景像装模块，不像改系统

## 10. 第一期建设范围

- 商户AI：知识问答、问题分类、处理建议
- 运营支持：调额审核、入网审核、变更审核辅助
- 数据支持：直营销售数据助手、固定报表辅助
- 平台底座：权限、审批、审计、知识检索、基础监控
- 目标是先跑通闭环，而不是一次做全

## 11. 第一期边界

- 不追求全业务域一次性覆盖
- 不做完全自治的多 Agent 协作
- 不做大规模自动写操作闭环
- 不做过度开放的自由数据分析
- 不替代人工审批，只做智能辅助和提效

## 12. 演进路线图

- 阶段一：MVP验证，跑通商户AI、运营支持、数据支持代表场景
- 阶段二：平台扩展，接入代理商AI和更多审核流程
- 阶段三：智能运营平台，实现场景配置化接入和平台化运营

## 13. 预期价值

- 提升知识获取和问题处理效率
- 提高审核一致性和标准化程度
- 缩短数据查询和报表产出时间
- 沉淀业务规则、案例和流程资产
- 为后续更多业务智能化提供统一底座

## 14. 项目成功关键点

- 场景选择要聚焦，先做高价值闭环
- 平台底座必须先把权限、审批、审计打牢
- Skill、Tool、Workflow、Knowledge 必须分层治理
- 业务规则要持续沉淀和迭代
- 评测和反馈机制要从第一期开始建设

## 15. 汇报收口页

- 第一阶段目标不是做“大而全”的AI系统
- 而是建设一个可落地、可扩展、可治理的收单AI平台
- 先形成平台底座和代表性场景闭环
- 再逐步扩展到更多业务域和更复杂的智能服务能力
